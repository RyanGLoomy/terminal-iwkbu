PENJELASAN SEQUENCE DIAGRAM
Sistem Integrasi Terminal IWKBU

1. Register PO

Diagram ini menggambarkan proses pendaftaran akun Pengusaha Otobus (PO) baru ke dalam sistem. Terdapat empat partisipan: Operator PO sebagai aktor, Browser, Supabase Auth sebagai layanan otentikasi, dan Database. Proses dimulai ketika Operator PO mengisi formulir registrasi yang berisi nama perusahaan PO, alamat email, dan kata sandi melalui Browser. Setelah formulir dikirim, Browser memanggil fungsi signUp milik Supabase Auth dengan menyertakan email, kata sandi, serta metadata peran (role) bernilai "po". Supabase Auth kemudian membuat akun pengguna baru dan mengembalikan token sesi. Setelah akun berhasil dibuat, sistem menyisipkan satu baris data baru ke dalam tabel po pada Database dengan nilai status_verifikasi diisi "menunggu", yang berarti PO tersebut belum diverifikasi oleh Staf IW. Apabila penyisipan data berhasil, Browser menampilkan halaman konfirmasi yang memberi tahu Operator PO bahwa pendaftaran telah diterima dan sedang menunggu proses verifikasi.

2. Login

Diagram ini menggambarkan proses masuk (login) ke dalam sistem bagi seluruh peran pengguna. Terdapat enam partisipan: Pengguna sebagai aktor, Browser, Proxy (middleware rute), API /api/auth/login, Supabase Auth, dan Database. Proses bermula saat Pengguna memasukkan alamat email dan kata sandi pada formulir login di Browser. Browser kemudian mengirimkan permintaan POST ke rute /api/auth/login yang berisi kredensial tersebut. API login meneruskan permintaan ke Supabase Auth melalui fungsi signInWithPassword untuk melakukan verifikasi otentikasi. Apabila kredensial valid, Supabase Auth mengembalikan token sesi kepada API. Selanjutnya, API melakukan query ke Database untuk mengambil data profil pengguna dan peran (role) dari tabel profiles dan user_roles. Setelah data profil dan peran diperoleh, API mencatat aktivitas login melalui pemanggilan log_activity dengan aksi "LOGIN". Akhirnya, API mengembalikan rute default dan peran pengguna kepada Browser beserta cookie sesi, lalu Browser mengalihkan Pengguna ke halaman dashboard yang sesuai dengan perannya.

3. Lupa Password

Diagram ini menggambarkan proses pemulihan kata sandi bagi pengguna yang lupa. Terdapat enam partisipan: Pengguna sebagai aktor, Browser, API /api/auth/forgot-password, Supabase Auth, layanan Email, dan API /api/auth/reset-password. Proses diawali ketika Pengguna menautkan tautan "Lupa Password" pada halaman login. Browser mengirimkan permintaan POST berisi alamat email ke rute /api/auth/forgot-password. API kemudian memanggil fungsi resetPasswordForEmail milik Supabase Auth, yang berfungsi memicu pengiriman email berisi tautan reset kata sandi ke alamat email yang dimasukkan. Setelah email terkirim, Browser menampilkan pesan bahwa tautan reset telah dikirim. Pengguna kemudian membuka emailnya dan menautkan tautan reset yang tersedia, yang mengarahkan Browser ke halaman /reset-password. Pada halaman tersebut, Pengguna memasukkan kata sandi baru. Browser mengirimkan permintaan POST ke rute /api/auth/reset-password yang selanjutnya memanggil fungsi updateUserById milik Supabase Auth untuk memperbarui kata sandi pengguna. Setelah kata sandi berhasil diubah, Browser menampilkan pesan konfirmasi dan mengalihkan Pengguna kembali ke halaman login.

4. Logout

Diagram ini menggambarkan proses keluar (logout) dari sistem. Terdapat empat partisipan: Pengguna sebagai aktor, Browser, API /api/auth/logout, dan Supabase. Proses dimulai ketika Pengguna menautkan menu "Logout" pada sidebar atau bilah navigasi. Browser menampilkan dialog konfirmasi yang meminta Pengguna untuk memastikan bahwa ia ingin keluar. Setelah Pengguna mengonfirmasi, Browser mengirimkan permintaan POST ke rute /api/auth/logout. API kemudian memanggil fungsi signOut milik Supabase untuk mengakhiri sesi pada sisi server. Setelah sesi diakhiri, API mencatat aktivitas logout melalui log_activity dengan aksi "LOGOUT", menghapus cookie sesi dari Browser, dan mengalihkan Pengguna kembali ke halaman login.

5. Kelola Profil

Diagram ini menggambarkan proses pembaruan data profil pengguna. Terdapat empat partisipan: Pengguna sebagai aktor, Browser, API /api/profile, dan Database. Proses diawali ketika Pengguna membuka menu profil. Browser menampilkan formulir yang berisi data profil saat ini seperti nama lengkap. Pengguna mengedit data yang diinginkan lalu menekan tombol "Simpan". Browser menampilkan dialog konfirmasi perubahan. Setelah Pengguna mengonfirmasi, Browser mengirimkan permintaan PATCH ke rute /api/profile yang berisi field yang berubah, misalnya full_name. API kemudian menjalankan query UPDATE pada tabel profiles di Database untuk memperbarui data. Setelah pembaruan berhasil, API mencatat aktivitas melalui log_activity dengan aksi "UPDATE_USER". Browser menerima respons bahwa profil telah diperbarui dan menampilkan notifikasi sukses kepada Pengguna.

6. Update Data Armada

Diagram ini menggambarkan proses pembaruan status armada oleh PO. Terdapat empat partisipan: Operator PO sebagai aktor, Browser, API /api/po/armada/[id], dan Database. Proses bermula saat Operator PO memilih salah satu armada yang terdaftar dan mengubah nilai status_operasionalnya, misalnya dari "aktif" menjadi "rusak". Browser mengirimkan permintaan PATCH ke rute /api/po/armada/[id] yang berisi nilai status baru. API pertama-tama melakukan query SELECT untuk memastikan armada tersebut memang dimiliki oleh PO yang sedang login. Setelah kepemilikan terverifikasi, API menjalankan query UPDATE pada tabel armada untuk mengubah status. Aktivitas ini kemudian dicatat melalui log_activity dengan aksi "UPDATE_ARMADA". Browser menerima respons bahwa status telah diperbarui dan menampilkan notifikasi sukses.

7. Hasil Rekonsiliasi Armada PO

Diagram ini menggambarkan proses PO melihat hasil rekonsiliasi armadanya. Terdapat tiga partisipan: Operator PO sebagai aktor, Browser, dan Database. Proses dimulai ketika Operator PO membuka menu Hasil Rekonsiliasi. Browser melakukan query ke Database untuk mengambil data dari tabel iwkbu_sync_status yang difilter berdasarkan po_id milik PO tersebut. Database mengembalikan daftar armada beserta status kepatuhan IWKBU masing-masing, yang mencakup informasi seperti compliant, non_compliant, atau pending. Browser menampilkan data tersebut dalam bentuk tabel status kepatuhan. Apabila Operator PO menautkan salah satu armada untuk melihat detail, Browser melakukan query tambahan ke Database untuk mengambil data findings (temuan) yang terkait dengan armada tersebut. Database mengembalikan detail temuan beserta riwayat klarifikasi, dan Browser menampilkan halaman detail lengkap dengan tombol "Tanggapi" bagi setiap temuan yang masih aktif.

8. Tanggapi Temuan / Klarifikasi

Diagram ini menggambarkan proses PO mengirim klarifikasi atas temuan yang diberikan oleh Staf IW. Terdapat lima partisipan: Operator PO sebagai aktor, Browser, API /api/po/findings/[id]/clarifications, Storage Bucket, dan Database. Proses diawali ketika Operator PO menautkan tombol "Tanggapi" pada salah satu temuan. Browser menampilkan formulir klarifikasi yang berisi kolom pesan atau alasan serta opsi untuk mengunggah berkas bukti pendukung. Operator PO mengisi alasan klarifikasi dan mengunggah berkas bukti jika tersedia. Browser mengirimkan permintaan POST berbentuk FormData ke rute /api/po/findings/[id]/clarifications yang memuat pesan klarifikasi dan berkas bukti. API pertama-tama mengunggah berkas bukti ke Storage Bucket dan mendapatkan kembali jalur berkas (file_path). Setelah unggahan berhasil, API menyisipkan baris baru ke dalam tabel finding_clarifications di Database beserta jalur berkas bukti. API juga memperbarui status temuan menjadi "on_progress" pada tabel findings. Aktivitas ini dicatat melalui log_activity dengan aksi "KIRIM_KLARIFIKASI". Browser menerima respons bahwa klarifikasi telah terkirim dan menampilkan notifikasi sukses kepada Operator PO.

9. Buka Sesi Petugas

Diagram ini menggambarkan proses pembukaan sesi kerja oleh Petugas Loket Terminal sebelum melakukan pencatatan transaksi kendaraan. Terdapat lima partisipan: Petugas Loket sebagai aktor, Browser, API /api/auth/verify-pin, API /api/sesi/open, dan Database. Proses dimulai ketika Petugas Loket menautkan tombol "Mulai Sesi" pada halaman loket. Browser menampilkan formulir yang meminta Petugas Loket untuk memilih nama petugas terminal dan memasukkan nomor PIN. Setelah PIN dimasukkan, Browser mengirimkan permintaan POST ke rute /api/auth/verify-pin yang berisi nilai PIN tersebut. API verify-pin melakukan query ke tabel petugas_terminal pada Database untuk mengambil daftar petugas yang aktif pada terminal tersebut. Setelah daftar diperoleh, API membandingkan PIN yang dimasukkan dengan nilai hash PIN yang tersimpan menggunakan fungsi bcrypt.compare. Apabila PIN cocok, API menyisipkan atau memperbarui sesi PIN pada tabel petugas_pin_sessions dengan masa berlaku delapan jam. API mencatat aktivitas "VERIFIKASI_PIN" dan mengembalikan hasil verifikasi beserta ID petugas dan data sesi kepada Browser. Setelah verifikasi PIN berhasil, Browser mengirimkan permintaan POST ke rute /api/sesi/open. API sesi/open pertama-tama memeriksa apakah masih ada sesi aktif yang belum ditutup. Apabila tidak ada, API menyisipkan baris baru ke tabel sesi_petugas dengan status "aktif" dan mencatat aktivitas "BUKA_SESI". API mengembalikan ID sesi kepada Browser, yang kemudian membuka menu pencatatan transaksi kendaraan untuk Petugas Loket.

10. Catat Kendaraan Masuk/Keluar Terminal

Diagram ini menggambarkan proses pencatatan transaksi kendaraan masuk oleh Petugas Loket Terminal. Terdapat empat partisipan: Petugas Loket sebagai aktor, Browser, API /api/transaksi/masuk, dan Database. Proses bermula saat Petugas Loket memasukkan nomor polisi kendaraan pada formulir pencatatan. Browser mengirimkan permintaan POST ke rute /api/transaksi/masuk yang berisi ID sesi aktif, ID PO pemilik armada, dan nomor polisi yang dimasukkan. API pertama-tama memvalidasi bahwa sesi PIN Petugas Loket masih berlaku dan bahwa sesi kerja aktif masih berjalan. Setelah validasi lolos, API melakukan query ke tabel armada untuk mencari armada dengan nomor polisi yang cocok menggunakan pencocokan ilehuruf besar-kecil (ILIKE). Database mengembalikan data armada yang cocok. API kemudian menyisipkan baris baru ke tabel kendaraan_masuk untuk mencatat transaksi masuk. Aktivitas ini dicatat melalui log_activity dengan aksi "INPUT_TRANSAKSI". Browser menerima respons bahwa transaksi telah tercatat dan menampilkan notifikasi sukses.

11. Lihat Rekap Dan Riwayat Data Terminal

Diagram ini menggambarkan proses Admin Terminal atau Petugas Loket melihat rekap dan riwayat transaksi terminal. Terdapat empat partisipan: Admin/Loket sebagai aktor, Browser, Proxy (middleware), dan Database. Proses diawali ketika aktor membuka menu Rekap dan Riwayat. Browser mengirim permintaan ke Proxy yang kemudian meneruskan ke rute /admin-terminal/rekap. API memeriksa peran aktor untuk menentukan cakupan data yang boleh diakses: Admin Terminal dapat melihat data secara global untuk seluruh terminal, sedangkan Petugas Loket hanya dapat melihat data dari sesinya sendiri. Setelah pengecekan peran, API memanggil stored procedure get_admin_rekap_harian pada Database untuk memperoleh data rekap harian. Database mengembalikan data rekap yang mencakup jumlah transaksi masuk, transaksi keluar, dan total nominal per hari. Browser menampilkan data tersebut dalam bentuk tabel. Aktor dapat menggunakan filter berdasarkan rentang tanggal atau plat nomor kendaraan. Browser mengirim permintaan baru dengan parameter filter, dan API melakukan query ulang dengan filter yang ditentukan. Database mengembalikan data terfilter, dan Browser memperbarui tabel.

12. Manajemen Akun Terminal

Diagram ini menggambarkan proses Admin Terminal menambahkan akun Petugas Loket baru. Terdapat lima partisipan: Admin Terminal sebagai aktor, Browser, API /api/admin/petugas, Supabase Auth dengan admin API, dan Database. Proses dimulai ketika Admin Terminal menautkan tombol "Tambah Petugas". Browser menampilkan formulir data petugas yang berisi nama petugas, alamat email opsional, dan kata sandi opsional. Setelah Admin Terminal mengisi formulir dan menekan tombol "Simpan", Browser mengirimkan permintaan POST ke rute /api/admin/petugas. API memanggil fungsi createUser milik Supabase Auth admin API untuk membuat akun otentikasi baru. Supabase Auth mengembalikan UUID pengguna yang baru dibuat. API kemudian menyisipkan data profil baru ke tabel profiles dan menetapkan peran "loket" pada tabel user_roles. Aktivitas ini dicatat melalui log_activity dengan aksi "BUAT_USER". Browser menerima respons bahwa petugas telah dibuat dan memperbarui tabel daftar petugas.

13. Manajemen Akun (Staf IW)

Diagram ini menggambarkan proses Staf IW mengelola akun pengguna, baik akun biasa (PO dan Loket) maupun akun Admin Terminal. Terdapat enam partisipan: Staf IW sebagai aktor, Browser, API /api/staf-iw/users, API /api/staf-iw/users/[id], API /api/staf-iw/admin-terminal, dan Database. Pada bagian pertama, Staf IW membuka menu Manajemen Akun. Browser mengirim permintaan GET ke rute /api/staf-iw/users. API melakukan query ke Database untuk mengambil data dari tabel profiles yang bergabung (join) dengan tabel user_roles. Database mengembalikan daftar pengguna beserta perannya masing-masing. Browser menampilkan daftar tersebut dalam bentuk tabel. Apabila Staf IW ingin mengubah peran seorang pengguna, ia memilih pengguna tersebut dan memilih peran baru. Browser mengirimkan permintaan PATCH ke rute /api/staf-iw/users/[id] yang berisi peran baru. API menghapus peran lama dan menyisipkan peran baru pada tabel user_roles. Aktivitas ini dicatat melalui log_activity dengan aksi "UPDATE_USER". Browser menampilkan notifikasi bahwa peran telah diperbarui. Pada bagian kedua, Staf IW membuka tab khusus untuk mengelola akun Admin Terminal dan Staf IW. Browser mengirim permintaan GET ke rute /api/staf-iw/admin-terminal. API melakukan query ke Database untuk mengambil profil pengguna dengan peran admin-terminal atau staf-iw. Browser menampilkan daftar akun admin tersebut. Apabila Staf IW ingin membuat akun admin baru, ia mengisi formulir berisi email, kata sandi, dan terminal yang ditugaskan. Browser mengirimkan permintaan POST ke rute /api/staf-iw/admin-terminal. API memanggil fungsi createUser milik Supabase Auth dan menyisipkan data profil beserta penetapan peran ke Database.

14. Kelola Master Data

Diagram ini menggambarkan proses Staf IW mengelola data master termasuk daftar terminal dan jenis kendaraan. Terdapat empat partisipan: Staf IW sebagai aktor, Browser, API /api/admin/terminals, dan Database. Proses diawali ketika Staf IW membuka menu Master Data. Browser mengirim permintaan GET untuk mengambil daftar terminal dan jenis kendaraan dari Database. API melakukan query SELECT pada tabel terminals dan tabel jenis_kendaraan. Database mengembalikan daftar lengkap data master tersebut. Browser menampilkan data dalam bentuk tabel. Staf IW dapat melakukan penambahan, pengubahan, atau penghapusan data terminal maupun jenis kendaraan. Setiap aksi dikirim melalui permintaan POST, PATCH, atau DELETE ke API. API menjalankan query INSERT, UPDATE, atau DELETE yang sesuai pada Database dan mencatat aktivitas melalui log_activity. Setelah operasi berhasil, Browser memperbarui tabel daftar master data.

15. Manajemen Data Armada

Diagram ini menggambarkan proses Staf IW mengimpor data kepatuhan IWKBU dari sumber eksternal. Terdapat lima partisipan: Staf IW sebagai aktor, Browser, API /api/staf-iw/iwkbu-source, IWKBU API eksternal, dan Database. Proses dimulai ketika Staf IW memilih metode pengambilan data kepatuhan: melalui sinkronisasi API langsung atau melalui unggahan berkas. Apabila Staf IW memilih metode unggahan berkas, Browser menampilkan antarmuka unggah berkas berformat CSV atau JSON. Staf IW mengunggah berkas yang berisi data kepatuhan IWKBU. Browser mengirimkan berkas tersebut sebagai FormData ke rute /api/staf-iw/iwkbu-source. API memproses berkas dengan mengurai dan menormalisasi nomor polisi yang terdapat di dalamnya. Setelah data dinormalisasi, API melakukan operasi UPSERT pada tabel iwkbu_source_records di Database, yaitu menyisipkan data baru atau memperbarui data yang sudah ada berdasarkan referensi eksternal. Aktivitas ini dicatat melalui log_activity dengan aksi "IMPORT_IWKBU". Browser menerima respons yang menyatakan jumlah baris yang berhasil diimpor dan menampilkan notifikasi sukses.

16. Proses Rekonsiliasi Data

Diagram ini menggambarkan proses sinkronisasi dan rekonsiliasi data antara sistem terminal dengan data IWKBU eksternal. Terdapat enam partisipan: Staf IW sebagai aktor, Browser, API /api/staf-iw/iwkbu-sync, fungsi executeIwkbuSync, IWKBU API eksternal, dan Database. Proses bermula ketika Staf IW menautkan tombol "Jalankan Sync". Browser mengirimkan permintaan POST ke rute /api/staf-iw/iwkbu-sync. API memanggil fungsi executeIwkbuSync dengan parameter trigger bernilai "manual". Fungsi ini pertama-tama menyisipkan baris baru pada tabel iwkbu_sync_runs dengan status "running" untuk mencatat bahwa proses sinkronisasi sedang berlangsung. Selanjutnya, fungsi mengirimkan permintaan POST ke IWKBU API eksternal dengan daftar nomor polisi armada untuk memeriksa status kepatuhan. IWKBU API mengembalikan data kepatuhan per armada. Fungsi kemudian menyimpan data tersebut ke dalam tabel iwkbu_source_records melalui operasi UPSERT. Setelah itu, fungsi mengambil data armada dan status PO dari Database untuk dibandingkan dengan data IWKBU. Hasil perbandingan disimpan dalam tabel iwkbu_sync_status. Akhirnya, fungsi memperbarui baris iwkbu_sync_runs dengan ringkasan hasil. API mengembalikan jumlah armada yang berhasil disinkronkan dan jumlah isu yang ditemukan. Browser menampilkan ringkasan hasil rekonsiliasi kepada Staf IW.

17. Hasil Rekonsiliasi Data

Diagram ini menggambarkan proses Staf IW melihat hasil rekonsiliasi secara menyeluruh. Terdapat empat partisipan: Staf IW sebagai aktor, Browser, API /api/staf-iw/rekonsiliasi, dan Database. Proses diawali ketika Staf IW membuka menu Hasil Rekonsiliasi. Browser mengirim permintaan GET ke rute /api/staf-iw/rekonsiliasi. API melakukan query ke Database untuk mengambil data PO berstatus aktif, PO berstatus menunggu verifikasi, serta data armada. Database mengembalikan data tersebut dalam bentuk ringkasan. Browser menampilkan tabel hasil rekonsiliasi yang menampilkan status kepatuhan IWKBU setiap armada. Staf IW dapat menggunakan filter pencarian berdasarkan nama atau status untuk mempersempit data yang ditampilkan. Browser mengirim permintaan GET baru dengan parameter pencarian dan status yang dipilih. API melakukan query ulang dengan filter tersebut, dan Database mengembalikan data yang telah terfilter. Browser memperbarui tabel dengan data hasil filter.

18. Kelola Tindak Lanjut Temuan

Diagram ini menggambarkan proses Staf IW menindaklanjuti temuan yang telah diklarifikasi oleh PO. Terdapat empat partisipan: Staf IW sebagai aktor, Browser, API /api/staf-iw/findings, dan Database. Proses dimulai ketika Staf IW membuka daftar temuan yang berstatus menunggu peninjauan. Browser mengirim permintaan GET ke rute /api/staf-iw/findings. API melakukan query ke Database dengan menggabungkan tabel findings dengan tabel po dan tabel armada untuk memperoleh konteks lengkap setiap temuan. Database mengembalikan daftar temuan beserta data PO dan armada terkait. Browser menampilkan temuan-temuan tersebut dalam bentuk tabel lengkap dengan bukti klarifikasi yang dikirim oleh PO. Staf IW meninjau bukti klarifikasi tersebut, kemudian memutuskan untuk menanggapi dengan menerima atau menolak. Browser mengirim permintaan PATCH yang berisi status keputusan dan catatan resolusi. API memperbarui status temuan pada tabel findings, mengisi kolom resolved_by dengan ID Staf IW, dan mengirim notifikasi kepada PO terkait hasil klarifikasi. Aktivitas ini dicatat melalui log_activity dengan aksi "UPDATE_TEMUAN". Browser menampilkan notifikasi sukses bahwa temuan telah diperbarui.

19. Generate Laporan Dan Ekspor

Diagram ini menggambarkan proses pembuatan dan ekspor laporan transaksi terminal. Terdapat empat partisipan: Admin Terminal atau Staf IW sebagai aktor, Browser, API /api/admin/reports/terminal, dan Database. Proses diawali ketika aktor membuka menu Laporan. Browser menampilkan formulir filter yang berisi pilihan rentang tanggal dan terminal. Setelah aktor mengatur filter dan menekan tombol "Tampilkan", Browser mengirim permintaan GET ke rute /api/admin/reports/terminal dengan parameter tanggal mulai, tanggal akhir, dan terminal yang dipilih. API memanggil stored procedure get_admin_rekap_harian pada Database untuk memperoleh data rekap per PO, per petugas, per armada, dan per hari. Database mengembalikan data rekap tersebut. Browser menampilkan pratinjau data dalam bentuk tabel. Apabila aktor ingin mengekspor laporan, ia menautkan tombol "Export Excel" atau "Export PDF". Browser membuat berkas laporan dengan format yang dipilih menggunakan pustaka xlsx atau jsPDF secara lokal di sisi klien. Berkas kemudian diunduh oleh Browser.

20. Audit Trail

Diagram ini menggambarkan proses Staf IW melihat jejak audit aktivitas sistem. Terdapat empat partisipan: Staf IW sebagai aktor, Browser, API /api/staf-iw/audit-trail, dan Database. Proses dimulai ketika Staf IW membuka menu Audit Trail. Browser mengirim permintaan GET ke rute /api/staf-iw/audit-trail dengan parameter rentang tanggal, jenis aksi, dan kata kunci pencarian. API memanggil stored procedure get_activity_logs pada Database dengan parameter filter yang diberikan. Database mengembalikan entri log aktivitas yang bergabung dengan tabel profiles untuk menampilkan nama pengguna yang melakukan aksi. Browser menerima data log beserta informasi apakah masih ada data lebih lanjut (hasMore). Browser menampilkan log aktivitas dalam bentuk tabel yang mencakup waktu, nama pengguna, jenis aksi, dan deskripsi. Staf IW dapat mempersempit data menggunakan filter jenis aksi, pencarian kata kunci, atau rentang tanggal. Browser mengirim permintaan GET baru dengan parameter filter yang diperbarui, dan API melakukan query ulang melalui stored procedure. Database mengembalikan data terfilter, dan Browser memperbarui tabel.

21. Notifikasi

Diagram ini menggambarkan proses pengiriman dan penampilan notifikasi kepada pengguna secara real-time. Terdapat lima partisipan: System Event sebagai pemicu, Database, Supabase Realtime, Browser, dan Pengguna sebagai aktor penerima. Proses dipicu oleh sebuah System Event, misalnya ketika Staf IW memperbarui status temuan menjadi "closed" atau "pelanggaran valid". Sistem memanggil fungsi createNotification dengan menyertakan ID pengguna penerima, judul, dan pesan notifikasi. Database menyisipkan baris baru pada tabel notifications. Setelah data tersimpan, Supabase Realtime menyiarkan (broadcast) peristiwa tersebut secara real-time melalui koneksi WebSocket. Browser yang sedang aktif menerima siaran tersebut dan mendorong notifikasi ke antarmuka pengguna, termasuk menampilkan badge notifikasi yang menandakan adanya pesan baru. Pengguna menautkan notifikasi tersebut. Browser mengirim permintaan UPDATE untuk menandai notifikasi sebagai telah dibaca (is_read = true) pada Database. Browser kemudian mengalihkan Pengguna ke tautan yang terkait dengan notifikasi tersebut, misalnya halaman detail temuan.
